## 4.1 为什么需要 Token 化

### 文本到数字的桥梁

```
模型只能处理数字，不能直接处理文本

"你好世界" 
    ↓ Token化
[token_1, token_2, token_3, ...]
    ↓ 查词表
[8876, 1234, 5678, ...]
    ↓ Embedding
[[0.1, 0.2, ...], [0.3, 0.1, ...], ...]

模型处理的是向量，不是文字
```

### 词汇表与 OOV 问题

```
方案1：以词为单位
  词表: ["你好", "世界", "中国", ...]
  问题: 遇到新词怎么办？（OOV: Out of Vocabulary）
  
方案2：以字符为单位
  词表: ["a", "b", "c", ..., "你", "好", ...]
  问题: 序列太长，丢失语义

方案3：子词切分（现代方案）
  词表: ["你", "好", "世", "界", "##ing", ...]
  优点: 平衡词表大小和序列长度
```

## 4.2 主流 Token 化算法

### BPE（Byte Pair Encoding）

```
BPE 训练过程

初始: ["l", "o", "w", "e", "r", ...]（单字符）

统计最频繁的相邻对，合并：
Step 1: "lo" 出现最多 → 合并为 "lo"
Step 2: "low" 出现最多 → 合并为 "low"
Step 3: "er" 出现最多 → 合并为 "er"
...
重复直到达到目标词表大小

结果: ["low", "er", "low", "est", ...]

GPT 系列使用 BPE
```

### WordPiece

```
与 BPE 类似，但合并策略不同

BPE: 选择频率最高的相邻对
WordPiece: 选择能最大化语言模型概率的相邻对

使用 "##" 标记非词首子词:
  "playing" → ["play", "##ing"]

BERT 使用 WordPiece
```

### SentencePiece

```
特点：语言无关的分词

不需要预分词（空格切分）
可以处理任何语言

Llama、Qwen 等使用 SentencePiece
```

### 不同 Tokenizer 对比

```python
# 同一句话，不同 Tokenizer 的结果

text = "Hello, 你好世界!"

GPT-4 (tiktoken):
  ["Hello", ",", " 你好", "世界", "!"]
  → 5 tokens

Llama (SentencePiece):
  ["Hello", ",", "▁你", "好", "世", "界", "!"]
  → 7 tokens

中文通常消耗更多 tokens
```

## 4.3 Token 计算实践

### 使用 tiktoken

```python
import tiktoken

# GPT-4 使用的编码
enc = tiktoken.encoding_for_model("gpt-4")

text = "Hello, 你好世界！"
tokens = enc.encode(text)

print(f"文本: {text}")
print(f"Tokens: {tokens}")
print(f"Token数: {len(tokens)}")

# 解码回文本
decoded = enc.decode(tokens)
print(f"解码: {decoded}")
```

### 中英文 Token 差异

```
英文: 1 token ≈ 4 字符 ≈ 0.75 词
中文: 1 token ≈ 1-2 汉字

示例:
"The quick brown fox"     → 4 tokens
"敏捷的棕色狐狸"          → 7 tokens

中文消耗的 token 约为英文的 2-3 倍！
成本和上下文窗口都要注意
```

### Token 估算经验法则

| 内容类型 | 估算规则 |
|---------|---------|
| 英文文本 | 1 token ≈ 4 字符 |
| 中文文本 | 1 token ≈ 1.5 汉字 |
| 代码 | 变化大，符号多 |
| 1000 中文字 | ≈ 650-800 tokens |
| 1000 英文词 | ≈ 1300 tokens |

### 代码的 Token 特点

```python
# 代码符号会产生额外 tokens

def hello():      # "def", " hello", "(", "):"
    print("Hi")   # "   ", " print", "(", '"Hi"', ")"
    
# 缩进、括号、引号都是独立 token
# 代码通常比同长度文本消耗更多 token
```

## 4.4 上下文窗口

### 什么是上下文窗口

```
上下文窗口 = 模型能"看到"的最大 token 数

输入 + 输出 必须在窗口内

┌─────────────────────────────────────┐
│            上下文窗口                │
│  ┌─────────────┐ ┌───────────────┐ │
│  │   输入      │ │    输出       │ │
│  │ (你的问题)  │ │ (模型回答)    │ │
│  └─────────────┘ └───────────────┘ │
│        ↑               ↑           │
│    消耗 token      消耗 token      │
└─────────────────────────────────────┘
```

### 各模型上下文长度

| 模型 | 上下文窗口 | 约等于 |
|------|-----------|--------|
| GPT-3.5 | 16K | 12K 汉字 |
| GPT-4o | 128K | 100K 汉字 |
| Claude 3 | 200K | 150K 汉字 |
| Gemini 1.5 | 1M+ | 750K+ 汉字 |
| Llama 3 | 8K | 6K 汉字 |

### 长上下文的挑战

```
问题1: 计算复杂度
  Attention 复杂度 = O(n²)
  上下文翻倍 → 计算量翻4倍
  
问题2: 中间内容丢失 (Lost in the Middle)
  ┌─────────────────────────────────┐
  │ 开头   ← 关注度高               │
  │ 中间   ← 关注度低（容易遗漏）   │
  │ 结尾   ← 关注度高               │
  └─────────────────────────────────┘

问题3: 成本
  更多 token = 更高费用
```

### 长上下文最佳实践

```
1. 重要信息放开头或结尾
2. 不要塞无关内容
3. 考虑使用摘要压缩
4. 必要时用 RAG 替代长上下文
```

## 4.5 上下文窗口管理策略

### 滑动窗口

```
保留最新的对话，丢弃最早的

对话历史: [消息1, 消息2, 消息3, 消息4, 消息5, 消息6]
                                          └─窗口内─┘
                  └─────丢弃─────┘

优点：简单
缺点：丢失早期重要信息
```

### 摘要压缩

```
把长内容压缩成摘要

原始 (1000 tokens):
  "会议讨论了项目进度，张三汇报了前端开发完成80%，
   李四说后端API已全部完成，王五提出需要增加测试..."

摘要 (100 tokens):
  "会议要点：前端80%，后端完成，需增加测试"

节省 token 同时保留关键信息
```

### 检索增强（RAG）

```
不把全部内容放入上下文，而是检索相关部分

┌─────────────────────────────────────┐
│  知识库（可能有100万字）            │
│  ┌─────┐ ┌─────┐ ┌─────┐          │
│  │文档1 │ │文档2 │ │文档3 │ ...      │
│  └─────┘ └─────┘ └─────┘          │
└─────────────────────────────────────┘
          │
          ↓ 根据问题检索
          ↓
┌─────────────────────────────────────┐
│  上下文窗口（只有相关片段）          │
│  "问题 + 检索到的相关段落"          │
│  → 模型生成回答                     │
└─────────────────────────────────────┘
```

### 选择性上下文

```
根据任务选择性地包含信息

写代码时：只包含相关函数定义
写总结时：只包含核心段落
多轮对话：只保留与当前话题相关的历史

原则：只给模型需要的信息
```

## 本章小结

- ✅ Token 是模型处理文本的基本单位
- ✅ BPE/WordPiece/SentencePiece 是主流算法
- ✅ 中文消耗的 token 约为英文的 2-3 倍
- ✅ 上下文窗口 = 输入 + 输出的 token 上限
- ✅ 长上下文有"中间丢失"问题，需要策略管理
